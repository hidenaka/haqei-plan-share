HaQei / アニメと易経 ・ ショートproduce の真因

真の原因＝私のミスでなく、ハーネスの deadlock バグ

実コード検証済み（scripts/creative-harness.mjs）
ショートの produce ゲートはそもそも誰も通過できない構造でした。vertical ショートは過去1本も produce を通っていません（前回の sh40 も integrity_invalidated＝壊れて放置）。＝「10候補を頑張って選定すれば通る」のではなく、選定を全部やっても最後に必ず詰まるバグです。

バグの中身（1行で）

produce が run を ready にするには manifest.preflight が必要。しかし縦動画(shorts/tiktok)の分岐は台帳を書いて途中で return し、manifest.preflight を書く行に到達しない（creative-harness.mjs 〜4849行で return / 書く処理は〜4864行＝到達不能）。→ run は永遠に qualified 止まり → produce の assertVerticalCreativeReady が必ず throw。

ついでに判明した設計ミスマッチ

この縦3-runは「候補を複数出してAI審判が1つ選ぶ」方式。だがショートは私が台本を1本書く単著フロー。審判が候補の別案を選ぶと、確定台本(full_script.json)が上書きされて同一short_idが復旧不能になる危険もある。＝単著フローにこの方式は根本的に合っていない。
※ CLAUDE.md のショート品質ゲートは元々「fork独立採点≧90（short-artwork.json）＋shorts-qa」であって、この縦bon選定ではない。

直し方（コアのガバナンス改修・選んでください）

案B（推奨・CLAUDE.md準拠）：produce を壊れた縦3-runから切り離す。produce は「音声→レンダ」だけを行い、品質は既定どおり shorts-qa.py ＋ fork独立採点≧90（short-artwork.json） で担保（＝CLAUDE.mdに元々書いてある正しいゲート）。単著フローに合致・上書き事故なし・死んだコード経路を使わない。この案なら sh42 も安全に produce まで通せる。
案A：deadlock を塞ぐ（vertical publish 分岐で manifest.preflight を rendered-video 証跡付きで書く）。縦bon選定の仕組みは残すが、単著フローとのミスマッチ（上書き事故リスク）は残る。改修が深く重い。
ご判断：案B（fork≧90に一本化・推奨） ／ 案A（deadlockだけ塞ぐ）
案Bを選べば、私が produce ゲートを fork≧90＋shorts-qa に付け替え → sh42 を produce（音声〜レンダ）→ shorts-qa → fork採点 → 人間ゲートまで通します。以後の全ショートも同じ正しいルールで回ります。
