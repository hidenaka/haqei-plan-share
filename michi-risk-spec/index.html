02Stage1（7月）で作るもの

東京都ハッカソン向けMVP。1シナリオ完動を優先。

#機能中身
1リスク地図東京都工事＋ほこナビを地図表示。歩行者/車椅子/ベビーカーモードで歩道規制・迂回・段差を強調
2利用者別リスク変換同一規制を利用者別の注意度（低/中/高）に変換
3お散歩/実踏ルート記録
（業務の芯）		地図上でルートを引き、工事規制と重ねて危険箇所を自動ハイライト。メモ保存・共有リンク/簡易しおり出力
4データ出力GeoJSON / CSV / API

後段（時間が余れば）：AI自動入力（お知らせPDF→規制下書き）、影響スコアの精緻化。除外（将来）：車線レベル可視化・ルート上プッシュ通知・公共交通連携(Stage2)・当事者向けB2B2C・課金機能。

03技術スタックとデータ

層技術
フロント / 地図Cloudflare Pages + MapLibre GL JS + 国土地理院タイル
APICloudflare Workers + Hono
DBCloudflare D1（SQLite）
データ東京都 工事オープンデータ ＋ 国交省 ほこナビ 歩行空間ネットワークデータ

疎結合の肝：データ層を restrictions（工事規制）/ walkspace（ほこナビ）/ transit（公共交通・Stage1は空）に分離。ルート判定を純粋関数で切り出し、10月のODPT拡張は transit を差し込むだけで済む形にする。

04ガードレール

  • 誤情報/公式誤認：verification_status ＋ 免責常時表示「実際の通行可否は警察・道路管理者・現地標識の公式情報を確認」
  • データ鮮度：最終更新日時を明示。Stage1はオープンデータのスナップショット＋手動サンプルと明記
  • 公開項目：場所・期間・影響・問い合わせ先・更新日時に限定

この設計でOKか

  1. Stage1の4機能（リスク地図／利用者別変換／お散歩ルート記録／データ出力）でMVPを作る。
  2. OKなら実装計画（writing-plans）を作り、Week1のコア（プロジェクト雛形＋D1＋地図表示＋データ投入）から着手。
  3. 修正したい点（機能の増減・優先順位・命名など）があればここで。無ければ「GO」で実装計画に進む。
